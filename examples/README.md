# Examples

Ready-made inventories and the reports they produce, generated by the actual `prahari` CLI. Use them to see the tool's output before building your own.

| Example | Scenario | Highlights |
| --- | --- | --- |
| [retail-bank](retail-bank) | Small retail bank, 4 models | spreadsheet-as-model (Para 7(3)), GenAI, a **critical** overdue-validation finding (Para 33) |
| [nbfc](nbfc) | Middle-layer NBFC, 4 models | a **third-party** vendor model (Para 45–48), mixed tiers |
| [genai-chatbot](genai-chatbot) | Customer GenAI estate, 3 models | autonomy-driven high tier (Para 52), an **untiered** bot finding (Para 17/21) |

Run the guided walkthrough: [`scripts/demo.sh`](../scripts/demo.sh) — captured output in [`DEMO.md`](DEMO.md).

## retail-bank

A small retail bank with four models across types — an ML scorecard, a pricing **spreadsheet** (a model under Para 7(3)), a **GenAI** AML assistant, and an untiered statistical forecaster.

- [`inventory.json`](retail-bank/inventory.json) — the inventory (a plain JSON file).
- [`report.md`](retail-bank/report.md) — the examiner/board report generated from it.

Reproduce it:

```bash
npm run build
F=examples/retail-bank/inventory.json
node packages/mrm/dist/cli.js check  --file "$F"
node packages/mrm/dist/cli.js report --file "$F" --org "Demo Retail Bank" --out examples/retail-bank/report.md
```

`check` output (exits non-zero because of the critical finding — useful in CI):

```text
[CRITICAL] VALIDATION_REPORT_OVERDUE (Para 33) — Retail Credit Scorecard: Validation report not placed before the RMCB within three months.
[WARNING]  UNTIERED (Para 17, 21) — Branch Footfall Forecaster: Live model has no risk tier assigned.
```

What this example demonstrates:

- **Para 7(3):** a pricing spreadsheet is a first-class model.
- **Para 20 (non-offsetting):** the spreadsheet is *high* tier (materiality 3) despite low complexity — it is not averaged down.
- **Para 52:** the GenAI assistant is *high* tier partly from autonomy.
- **Para 33:** a high-tier model whose validation report is overdue raises a **critical** finding.
- **Para 17/21:** a live model with no tier is flagged.

## nbfc

A middle-layer NBFC with four models — an internal credit-underwriting model, a **third-party** income-estimation model bought from a vendor, a statistical collections ranker, and a KYC OCR model.

- [`inventory.json`](nbfc/inventory.json) — the inventory.
- [`report.md`](nbfc/report.md) — the generated report (clean: no open findings).

Reproduce it:

```bash
npm run build
F=examples/nbfc/inventory.json
node packages/mrm/dist/cli.js check  --file "$F"
node packages/mrm/dist/cli.js report --file "$F" --org "Demo NBFC (Middle Layer)" --out examples/nbfc/report.md
```

`check` exits zero — every model is tiered and its roles are populated, so there are no open findings. The interesting governance is structural, not a finding:

What this example demonstrates:

- **[Para 45–48](../reference/controls/para-45-48-third-party.md) (third-party):** *Income Estimation (vendor)* is flagged `isThirdParty: true`. It keeps a full RE validator (`vteam`) and approver — a vendor certificate does **not** excuse independent RE validation (Para 46), and the RE owner (`ravi`) stays accountable ([Para 8](../reference/controls/para-08-accountability.md), Para 45).
- **[Para 20](../reference/controls/para-17-20-risk-tiering.md) (non-offsetting):** the vendor model is *high* (materiality 3) even though complexity is 2 — the high factor wins.
- **[Para 21–22](../reference/controls/para-21-22-inventory.md):** all four models carry the four roles, tier, and intended use.
- A clean report is also a result: it is the evidence an examiner can read that the inventory is complete and tiered.

## genai-chatbot

A bank's customer-facing generative-AI estate — a high-autonomy customer-support copilot, a marketing-copy generator, and an internal policy Q&A bot that was deployed **without a risk tier**.

- [`inventory.json`](genai-chatbot/inventory.json) — the inventory.
- [`report.md`](genai-chatbot/report.md) — the generated report (one warning).

Reproduce it:

```bash
npm run build
F=examples/genai-chatbot/inventory.json
node packages/mrm/dist/cli.js check  --file "$F"
node packages/mrm/dist/cli.js report --file "$F" --org "Demo Bank — GenAI" --out examples/genai-chatbot/report.md
```

`check` output:

```text
[WARNING] UNTIERED (Para 17, 21) — Internal Policy Q&A Bot: Live model has no risk tier assigned.
```

What this example demonstrates:

- **[Para 52](../reference/controls/para-17-20-risk-tiering.md) (autonomy):** the *Customer Support Copilot* is *high* tier — its autonomy input (3) drives the tier up, exactly as Para 52 intends for AI models that act on their outputs.
- **[Para 54](../reference/controls/para-54-ai-behavioural-explainability.md):** these are the generative, behavioural-risk models that need explainability thresholds, adversarial testing, and hallucination guardrails — the copilot, being customer-facing, is held to the higher threshold.
- **[Para 59](../reference/controls/para-59-ai-deployment-security.md):** the customer-facing copilot is where prompt-injection controls, AI disclosure, and a human-handoff path are required.
- **[Para 17/21](../reference/controls/para-21-22-inventory.md):** the untiered internal bot is flagged — a live model with no tier is a governance gap, not a free pass.
